Mecanim vs. Animancer

This section compares the way each animation system affects your code architecture and development pipeline. If you're interested in the differences in performance between systems, take a look at the Performance Benchmarks section.

Unity’s inbuilt Mecanim animation system was designed to be easily accessible to people without much programming experience. It achieves that goal quite well, but doing so creates the illusion that animations should define the way your game works, when in fact the opposite is true.

This state diagram taken straight from the Unity Manual actually illustrates the problems with that approach quite effectively. To transition from the Walk state to the Fall state, a script must detect that the character is no longer on the ground and a transition must be set up between those states. Then when the script detects that the character is back on the ground it can try to go back to idle, except there is no transition from Fall to Idle so nothing will happen. Time was originally spent creating the state machine and the code logic separately, and now you have to waste even more time figuring out whether the problem is a bug in your code or in the state machine (or both) and then fixing it without breaking anything else.

That problem would never occur with Animancer. You still need to detect when the creature leaves and returns to the ground, but when you tell it to play an animation it will simply play that animation. If the animation doesn’t play, then you only need to look at your code to find the bug instead of examining multiple separate areas of your project.

The benefits of Animancer can be summarised in five key areas:

  • Adaptability - Animancer allows your scripts to define the states they will be interacting with in whatever way you find most logical for each situation instead of forcing you to define the structure of your state machine all in one place separately from your scripts like Mecanim.
  • Flexibility - Animancer can play any animation without any extra preparation instead of forcing you to define the exact structure of all your states, transitions, and parameters in the Unity Editor like Mecanim.
  • Safety - Animancer doesn’t require you to use magic strings so it avoids the common problems they cause when using Mecanim. No more time wasted fixing bugs caused by misspelling a state name or renaming something without knowing where you need to update that name in your code. You can still choose to use strings if you want to.
  • Clarity - Animancer makes it much clearer what each object needs and can do. If a script has a "Jump Animation" field then it needs a jump animation, or if an object doesn't have a jump script then it can't jump and doesn't need a jump animation. No more looking at a state without any clear indication of which scripts interact with it.
  • Reliability - Animancer is a slave system designed to do exactly what your scripts tell it to do instead of trying to manage its own behaviour like Mecanim. Animancer won’t simply ignore your attempt to start a transition because it isn't in the right state and won’t require you to check whether the animation you just told it to play is actually being played.
