Animancer States

When you play an animation, Animancer creates a ClipState to manage it and keep track of its progress. It also stores that state in a dictionary so that it can be efficiently reused whenever you want to play that same animation again.

There are 3 main overloads of the AnimancerController.Play method (and similar ones for CrossFade and CrossFadeNew):

  • Play(AnimationClip clip, int layerIndex = 0) uses the clip as the dictionary key to get or create its state on the specified layer.
  • Play(object key) gets a state that was previously registered with the key or returns null if none was present.
  • Play(AnimancerState state) plays the state directly, no keys involved. You can get the AnimancerState returned from any previous call to Play or GetOrCreateState. Note that calling this method on the AnimancerController will stop all other states on the same layer while calling state.Play will only play it without affecting any other states.

State Constructors

You can simple create a new ClipState or any other type that inherits from the base AnimancerState class such as AnimatorControllerState or AnimationMixer.

The AnimancerState types included in Animancer all have 3 common constructor types:

Parameters Description
AnimancerLayer layer Connects the state to a spare port on the layer or allocates a new port if none are spare. This constructor is commonly used by passing in an AnimancerController which uses an implicit cast to target its default layer. Otherwise, a specific layer can be retrieved using AnimancerController.GetLayer.
IAnimationMixer parent, int portIndex Connects the state to a specific port on the parent. This constructor is usually only used to connect a state to a specific port on an Animation Mixer. An AnimancerLayer can also be used as the parent, but layers should generally be allowed to manage their own ports using the constructor above.
PlayableGraph graph Creates the state without connecting it to anything so it will do nothing until you call SetParent.

You can also define your own Custom State Types.

Custom Keys

Animancer uses AnimationClips as their own dictionary keys by default, but it also allows for the flexibility of using any type of key you like:

  • The LegacyAnimancerController component overrides the GetKey method to use the clip's name instead of the AnimationClip itself. Calling Play(AnimationClip clip) uses the name to lookup the state, but you can also pre-register the clip on startup by adding it to the Animations array in the inspector or calling GetOrCreateState(clip) so that later on you can call Play("Animation Name") later on without a direct reference to the AnimationClip.
  • You can simply use any type of key with any of the methods that take an object key. For example, you might want to use a custom enum as a key. You could also mix and match key types if you want to have standard actions mapped to an enum with other once-off animations used for specific interactions or in a scripted cinematic used as their own keys.
  • You don't even need to register a state in the dictionary if you don't want to. You can simply create a new ClipState (or whatever type of state you want) without calling AnimancerPlayable.AddState.
  • If you want to have multiple states playing the same clip (such as if you need to play it on multiple layers) you will need to register them all with different keys (or not register them at all). CrossFadeNew uses this to fade a clip out while fading another copy of it in at the same time.
