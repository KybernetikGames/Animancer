01 Spider Bot

Location: Assets/Plugins/Animancer/Examples/02 Fine Control/01 Spider Bot

Namespace: Animancer.Examples.FineControl

This example demonstrates how you can make a robot that can wake up to move then go back to sleep by playing its awake animation backwards.

Pro-Only features are used in this example: modifying AnimancerState.Speed and AnimancerState.NormalizedTime. Animancer Lite allows you to try out these features in the Unity Editor, but they are not available in runtime builds. See the Feature Comparison for more information.

The Locomotion/Directional Blending example expands upon this concept to implement proper locomotion for the bot to move in any direction.

This example is implemented by the SpiderBotBasic script which looks like this (with the comments removed since we're about to explain how it works):

using Animancer;
using System;
using UnityEngine;

public sealed class SpiderBotBasic : MonoBehaviour
{
    [SerializeField] private AnimancerComponent _Animancer;
    [SerializeField] private AnimationClip _WakeUp;
    [SerializeField] private AnimationClip _Move;

    private Action _PauseGraph;
    private Action _FadeToMovement;

    private void Awake()
    {
        _Animancer.Play(_WakeUp);
        _Animancer.Playable.Evaluate();
        _Animancer.Playable.PauseGraph();

        _PauseGraph = _Animancer.Playable.PauseGraph;
        _FadeToMovement = () => _Animancer.CrossFade(_Move);
    }

    private void Update()
    {
        if (Input.GetKeyDown(KeyCode.Space))
        {
            var state = _Animancer.CrossFade(_WakeUp);
            state.Speed = 1;

            state.OnEnd = _FadeToMovement;

            _Animancer.Playable.UnpauseGraph();
        }

        if (Input.GetKeyUp(KeyCode.Space))
        {
            var state = _Animancer.CrossFade(_WakeUp);
            state.Speed = -1;

            if (state.Weight == 0 || state.NormalizedTime > 1)
                state.NormalizedTime = 1;

            state.OnEnd = _PauseGraph;
        }
    }
}

Initialisation

We want to make sure we start at the beginning of the SpiderBotWakeUp animation, but we don't actually want to do anything yet so we play it, then immediately call Evaluate to apply it to the model then pause the graph.

_Animancer.Play(_WakeUp);
_Animancer.Playable.Evaluate();
_Animancer.Playable.PauseGraph();

We could have used _Animancer.Play(_WakeUp).IsPlaying = false; or set the Speed = 0, but pausing the entire graph is better for performance because it means that Unity doesn't need to evaluate its output every frame.

Waking and Sleeping

The Update method for waking up and going back to sleep is quite straightforward:

private void Update()
{
    // Wake up and start moving when Space is pressed.
    if (Input.GetKeyDown(KeyCode.Space))
    {
        // Make sure the graph is unpaused (because we pause it when going back to sleep).
        _Animancer.Playable.UnpauseGraph();

        // Play the wake up animation forwards.
        var state = _Animancer.CrossFade(_WakeUp);
        state.Speed = 1;

        // When finished waking up, start moving.
        state.OnEnd = _FadeToMovement;
    }

    // Go back to sleep when Space is released.
    if (Input.GetKeyUp(KeyCode.Space))
    {
        // Play the wake up animation backwards.
        var state = _Animancer.CrossFade(_WakeUp);
        state.Speed = -1;

        // If it was not already playing, start at the end of the animation.
        // Otherwise just play backwards from the current time.
        if (state.Weight == 0 || state.NormalizedTime > 1)
            state.NormalizedTime = 1;

        // When finished going to sleep, pause the graph.
        state.OnEnd = _PauseGraph;
    }
}

Note that if we go back to sleep in the middle of waking up, we can just reverse the Speed. But after the _WakeUp animation ends and fades into _Move it gets automatically stopped and returns its Time to 0, so playing backwards from there would do nothing (it would immediately end). Instead, we need to make sure that the state.NormalizedTime = 1 to start it at the end if it wasn't already active (state.Weight == 0) or if it was past the end and still fading out (state.NormalizedTime > 1).

In this example, state.Weight == 0 achieves the same thing that could be done with !state.IsPlaying. However, in a more complex system the animation may have been paused to freeze it in place, in which case we would want it to play from its current Time rather than snapping it to the end of the animation just because it was paused. So we want to act depending on whether the animation is currently affecting the model (Weight), not simply on whether it is playing (IsPlaying).

Caching Delegates

Unfortunately, due to the way delegates work in C#, every time we assign a method to a delegate it allocates a new object in managed memory. In the case of the AnimancerState.OnEnd event, this means that a simple line like state.OnEnd = _Animancer.Playable.PauseGraph; would allocate a new object (which is fine) and then when the OnEnd event is cleared that object would need to be garbage collected (which is bad for performance). We can work around that by creating the delegate on startup, but unfortunately it increases the complexity of our code from that single line to:

// A field to store the delegate.
private Action _PauseGraph;

private void Awake()
{
    ...

    // Create the delegate on startup.
    _PauseGraph = _Animancer.Playable.PauseGraph;
}

private void Update()
{
    ...

    // Use the delegate we already created.
    state.OnEnd = _PauseGraph;
}

And the same thing again for _FadeToMovement.

In other situations, we could simplify the delegate caching by replacing the AnimationClip field with a ClipState.Serializable which contains an AnimationClip, some other details, and an OnEnd callback which does not automatically get cleared so it can assign the same one to the state every time it plays. Unfortunately, in this situation we want to reuse the same state going both ways with a different callback at each end, so we wouldn't be able to cache them both with only one serializable. We could make a single method which chooses what to do based on the current input, but that would add more complexity than simply caching the delegates manually.
